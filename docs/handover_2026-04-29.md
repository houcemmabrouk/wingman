## 5. Conventions / contraintes du projet

- **Politique 3-pour-1** (CLAUDE.md) : aucune nouvelle feature sans en
  remplacer 3. La v2 viole cette règle (10 ajouts vs 4 retraits).
- **Auth** : `AUTH_DISABLED=true` en dev. CORS limite à `localhost:3000`,
  `localhost:3001`, `wingman.veridis.shop` (à étendre pour 3002 si v3).
- **Hot-reload** : uvicorn `--reload-dir app` (édits des scripts root
  `generate_*.py` ne reload plus le backend). Si tu modifies content.py,
  reload se déclenche → tue les pipelines en cours.
- **Git cassé** : le `.git` est partiel (manque `objects/` et `refs/`). Pas
  d'historique. Toute revert manuelle nécessite mémoire/recherche.
- **Pas de comparaison user vs user** (manifeste KPI § IV).
- **Pas de XP / badges / levels** (idem) — d'où la mort proposée
  d'Achievements et Challenges.

## 7. Workflow de déploiement

```bash
# Rebuild + deploy v2 (3001) — ne touche pas v1 (3000)
docker compose -f docker-compose.yml build frontend
docker tag wingman-vps-frontend:latest wingman-vps-frontend:next
docker stop wingman-frontend-next && docker rm wingman-frontend-next
docker run -d --name wingman-frontend-next --restart unless-stopped \
    -p 3001:3000 wingman-vps-frontend:next
docker network connect wingman-vps_default wingman-frontend-next  # GOTCHA

# Promouvoir v2 → 3000 (quand prêt)
docker tag wingman-vps-frontend:variant-v2-2026-04-29 wingman-vps-frontend:latest
docker compose up -d frontend

# Restaurer v1 prod
docker tag wingman-vps-frontend:variant-v1-original wingman-vps-frontend:latest
docker compose up -d frontend
```
