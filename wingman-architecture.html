Formules de calcul

Definissez la logique de chaque metrique. Colonne gauche = implementation actuelle. Colonne droite = votre nouvelle formule.

M
LM Mastery (mastery par module)
Actuel
Implementation actuelle
// mastery.py → update_mastery() scores = SELECT score FROM performance_records WHERE user_id AND module_id ORDER BY created_at DESC LIMIT 5 mastery_level = AVG(scores) // UPSERT dans lm_mastery // Niveaux: strong (>=80), adequate (>=60), weak (<60)
Nouvelle formule proposee
Probleme identifie : AVG des 5 derniers scores ne tient pas compte de la granularite LOS, du niveau Bloom, ni du temps ecoule depuis la derniere tentative. Un etudiant qui rate 5 fois puis reussit 5 fois aura 100% de mastery.
performance_records lm_mastery learning_outcomes question_attempts
T
Topic Mastery (mastery par topic CFA)
Actuel
Implementation actuelle
// kpis.py → GET /api/kpis/topic-mastery topic_mastery = SELECT t.code, AVG(lm.mastery_level) AS mastery FROM lm_mastery lm JOIN learning_modules m ON lm.module_id = m.id JOIN topics t ON m.topic_id = t.id WHERE lm.user_id = :uid GROUP BY t.code // Tous les modules ont le meme poids dans le topic
Nouvelle formule proposee
Probleme identifie : AVG simple de tous les modules d'un topic. Un module avec 2 LOS pese autant qu'un module avec 15 LOS. Pas de ponderation par importance ou nombre de questions.
lm_mastery learning_modules topics learning_outcomes
G
Global Progression (barre topbar)
Actuel
Implementation actuelle
// Topbar.tsx → calcul frontend EXAM_WEIGHTS = { ETH: 15, QM: 10, ECO: 10, FSA: 13, CORP: 8, EQU: 11, FI: 11, DER: 6, ALT: 7, PM: 9 } global = SUM(topic_mastery[i] * weight[i]) / SUM(weight[i]) // Objectif: 70% (MPS CFA)
Nouvelle formule proposee
Probleme identifie : Depend directement de Topic Mastery (qui depend de LM Mastery). Si la base est fausse, tout est faux. De plus, un topic non etudie (mastery=0) tire la moyenne vers le bas de facon brutale.
lm_mastery topics.weight_pct EXAM_WEIGHTS (constante frontend)
S
SRS — Repetition espacee
Actuel
Implementation actuelle
// srs.py → update_srs() if score >= 80: // Succes ease = min(3.0, ease + 0.1) interval *= ease next_review = now + interval days elif score >= 60: // Adequat next_review = now + interval // pas de changement else: // Echec (< 60) ease = max(1.3, ease - 0.2) interval = 1 // reset a 1 jour next_review = now + 1 day // ease_factor: 1.3 → 3.0 // Inspire de SM-2 (SuperMemo)
Nouvelle formule proposee
Note : L'algorithme SM-2 est eprouve mais les seuils (60/80) sont arbitraires. Le score utilise est celui du module entier, pas de la carte individuelle.
srs_queue performance_records.score

R
Readiness Score (pret pour l'examen)
Actuel
Implementation actuelle
// kpis.py → readiness_score readiness = 0.35 * mastery_score // AVG(lm_mastery) + 0.25 * recent_avg_score // AVG des 10 derniers quiz + 0.15 * retention_score // SRS success rate + 0.15 * coverage_score // modules couverts / 57 + 0.10 * duration_score // heures totales normalisees // Poids: 35% mastery + 25% recent + 15% retention // + 15% coverage + 10% volume
Nouvelle formule proposee
Probleme identifie : Les poids (35/25/15/15/10) sont arbitraires. duration_score recompense le volume brut (un etudiant qui passe 100h sans progresser aura un bonus). coverage ne distingue pas "vu 1 fois" de "maitrise".
lm_mastery performance_records srs_queue sessions

Weakness Detection (detection faiblesses)
Actuel
Implementation actuelle
// weakness_detector.py if score >= 60: return // rien // Classification type = "conceptual" if score < 40 else "calculation" // Severite severity = 5 if score < 30 else 4 if score < 40 else 3 // Regression: prev mastery > 70 && new < 55 if prev_mastery > 70 and score < 55: INSERT alert "Regression"
